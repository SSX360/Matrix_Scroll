AP2 Vault Card · hardware preview

NFC tap-to-approve. Same verifier contract as software.

Matrix Scroll ships as software today — emulated Ed25519 signing on your machine. The AP2 Vault Card is the hardware trust upgrade: a deliberate physical tap captures human-present approval while the private key stays sealed in an SE050 secure element on nRF52840.

Shipping now: PyPI 0.2.6, Git hooks, Scroll Gate PR verification, browser verifier, and emulated-mode evaluation.
In progress: SSX360 SE050 hardware preview and verifier-compatible external Ed25519 signer guidance.
Not: IAM, sandboxing, prompt filtering, or an agent runtime.

Software → hardware transition

Emulated mode stores the signing key locally so teams can evaluate the full commit envelope flow today. Hardware mode keeps the same schema and verifier — but moves the key into the secure element so the host cannot export it.

NFC tap-to-approve

For payment mandates, a deliberate tap on the AP2 Vault Card captures explicit human approval at the moment money moves. Phase 1 pilots use an emulated tap via the Matrix Scroll CLI until NFC companion ships.

Reference stack

nRF52840

Low-power MCU with NFC-A front end for tap-to-approve UX and USB provisioning during fleet enrollment.

SE050 secure element

Non-exportable Ed25519 signing inside the chip boundary — the host requests signatures but cannot read the private key.

Same proof surfaces

CLI, browser verifier, and CI action stay unchanged. Hardware signs the same canonical manifest bytes Matrix Scroll already uses.

Why this path exists

Possession factor

A compromised agent session cannot trigger approval without the physical card present for tap authorization.

Bounded mandates

Per-cart, budget cap, TTL, and merchant scope — exceed the bounds and a fresh tap is required or the payment is refused.

Design-partner only

Preview access stays narrow until latency, provisioning, and acceptance criteria are documented against the existing verifier contract.
